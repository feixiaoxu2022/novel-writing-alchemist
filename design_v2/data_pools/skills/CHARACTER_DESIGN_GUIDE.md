# 人物设计指南

> 本文档指导如何设计characters.json中的人物体系。与SKILL.md中"人物刻画技巧"（写作阶段如何描写人物）不同，本文档聚焦**设计阶段如何构建人物**。

## 1. 主角traits设计

### 1.1 选择原则

traits是角色的性格锚点，决定了角色在故事中的一切行为逻辑。3-5个traits不是随意堆叠形容词，而是构建一个**有内在张力的性格系统**。

**错误示范**（同义词堆叠）：
```
traits: ["勇敢", "坚强", "无畏", "刚毅", "果敢"]
→ 五个词说的是同一件事，角色是一块铁板，无趣
```

**错误示范**（购物清单式罗列）：
```
traits: ["聪明", "善良", "幽默", "勤奋", "乐观"]
→ 完美人设 = 纸片人，读者记不住任何一个特点
```

**正确做法**（有张力的组合）：
```
traits: ["冷静理智", "内心孤独", "技术偏执", "嘴硬心软"]
→ 冷静vs孤独产生内在矛盾：越理智越不被理解，越孤独越用理智武装自己
→ 技术偏执+嘴硬心软产生行为反差：工作中是完美主义暴君，私下偷偷帮人
```

### 1.2 构建traits的三步法

**第一步：确定核心矛盾**

每个有吸引力的角色都有一个核心矛盾。先找到这个矛盾，再围绕它选traits。

| 核心矛盾类型 | 示例 | 适合的故事类型 |
|---|---|---|
| 外在强势 vs 内心脆弱 | 表面冷酷的杀手内心渴望温暖 | 虐心、救赎 |
| 理想主义 vs 现实妥协 | 相信正义的刑警不得不放走线人 | 悬疑、商战 |
| 渴望连接 vs 害怕背叛 | 想要朋友却总是推开所有人 | 甜爽、成长 |
| 天赋异禀 vs 自我怀疑 | 最强的战士不相信自己配得上胜利 | 竞技、冒险 |

**第二步：围绕矛盾选择3-5个traits**

矛盾确定后，traits应该分为两组：

- **表层traits**（别人能看到的）：2-3个
- **深层traits**（只有亲近的人才知道的）：1-2个

```
核心矛盾：渴望被爱 vs 害怕受伤
表层traits: ["桀骜不驯", "毒舌"]  ← 这是保护壳
深层traits: ["极度敏感", "重情义"]  ← 这是真实的她
```

**第三步：验证traits能否驱动行为**

每个trait都应该能回答："当遇到X情况时，这个角色会怎么做？"

```
trait "技术偏执" → 遇到bug时会连续72小时不睡觉直到解决
trait "嘴硬心软" → 嘴上说"关我什么事"，转身默默帮忙
trait "内心孤独" → 聚会时最先离开，回家对着空房间发呆
```

如果一个trait无法驱动任何具体行为，它就是废词，换掉。

## 2. 动机设计：显性 vs 隐性

### 2.1 两层动机的关系

显性动机（overt_motivation）是角色**说出来**的目标，隐性动机（covert_motivation）是角色**自己可能都没意识到**的深层需求。两者之间的**落差**制造角色深度。

| | 显性动机 | 隐性动机 |
|---|---|---|
| 定义 | 角色公开宣称的目标 | 真正驱动行为的深层需求 |
| 可见性 | 对话中直接说出 | 通过行为暗示，可能角色自己都不知道 |
| 在故事中的作用 | 推动外部剧情 | 推动内心成长弧 |

### 2.2 设计方法

**核心问题**：角色说要A，但真正渴望的其实是B。

```
角色：失忆的刺客
显性动机：找回记忆（他说的）
隐性动机：证明自己不是怪物（他真正需要的）
→ 故事张力：找回记忆后发现自己确实是怪物，这时显性目标达成了，但隐性需求被摧毁了
```

```
角色：想考上名校的高中生
显性动机：考上清华（她说的）
隐性动机：获得父亲的认可（她真正需要的）
→ 故事张力：考上了，但父亲依然冷漠，才发现考清华从来不是答案
```

### 2.3 动机与X轴模式的匹配

动机不是凭空设计的，必须与选定的X轴剧情模式对齐：

```
X轴=A3复仇的罪：
  显性动机应该指向"复仇/讨回公道"
  隐性动机应该指向"修复创伤/自我救赎"

X轴=D20献身的解脱：
  显性动机应该指向"保护某人/完成使命"
  隐性动机应该指向"逃避/赎罪/寻求解脱"
```

## 3. 秘密恐惧设计

### 3.1 恐惧的作用

secret_fear是角色最脆弱的地方，它的作用：

1. **制造高潮**：故事最精彩的时刻，就是角色不得不直面恐惧的时刻
2. **驱动成长弧**：克服恐惧 = 角色成长；屈服于恐惧 = 悲剧结局
3. **解释行为**：角色的很多"不合理"行为，其实都是在逃避恐惧

### 3.2 设计原则

**恐惧必须与隐性动机相关**：

```
隐性动机：证明自己不是怪物
秘密恐惧：发现自己真的就是怪物
→ 恐惧是隐性动机的反面，形成完美张力
```

**恐惧必须具体，不能抽象**：

```
❌ "害怕失败" → 太泛，谁都怕
✅ "害怕在所有队友面前失误，证实他们背后的议论是对的" → 具体到场景
```

```
❌ "害怕孤独" → 太泛
✅ "害怕有一天醒来发现身边的人都是因为利用价值才留下的" → 具体到情境
```

## 4. 配角设计

### 4.1 功能定位

配角不是"凑人数"，每个配角必须对主角的故事线有明确功能：

| 功能 | 作用 | 设计要点 |
|---|---|---|
| **助力** | 帮助主角达成目标 | 不能是万能的，助力方式要有代价或局限 |
| **阻碍** | 制造障碍和冲突 | 阻碍动机要合理，不能是纯粹的"坏人" |
| **反转点** | 关键时刻改变局面 | 反转必须有伏笔，不能突然变脸 |

### 4.2 命名要求

**所有角色必须有可在正文中识别的名字或固定称谓**，这是硬性要求。因为后续会验证characters.json中设计的每个人物是否都出现在大纲和正文中——如果角色只有泛称，就无法在正文中被识别。

```
❌ name: "系统管理员"  → 正文中谁都可能是系统管理员，无法唯一识别
❌ name: "副本BOSS（七个）" → 这不是一个角色，是一类实体
❌ name: "神秘人" → 太泛，正文中无法追踪

✅ name: "零号管理员" → 固定称谓，正文中可唯一识别
✅ name: "顾母" → 固定称谓，虽然不是名字，但正文中一致使用
✅ name: "陆离" → 正式名字
```

### 4.3 配角与主角的关系

配角的功能要通过与主角的**关系**来实现，而不是孤立存在：

```
❌ 配角"黑客小王" - 功能：助力 - 描述：帮主角破解系统
  → 这是个工具人，不是角色

✅ 配角"黑客小王" - 功能：助力 - 描述：主角的前室友，欠主角一个救命之恩，
   技术高超但有赌瘾，帮忙的同时可能因赌债惹上麻烦
  → 有自己的弱点和故事线，助力不是无条件的
```

## 5. 人物关系设计（relationship_dynamics）

### 5.1 核心原则

relationship_dynamics不是简单列出"A和B是朋友"，而是描述**关系的变化轨迹**：

```
❌ "林羽和苏浅是恋人关系"
  → 静态描述，对创作没有指导意义

✅ "林羽和苏浅初期互相试探（不信任），中期因共同危机建立依赖（被迫同盟），
   后期发现彼此的秘密后经历信任危机（背叛感），最终选择坦诚面对（和解）"
  → 四个阶段的变化轨迹，直接对应大纲的四个转折点
```

### 5.2 设计方法

**关系变化三要素**：

1. **初始状态**：两人关系的起点（陌生/敌对/暧昧/旧识重逢）
2. **张力来源**：什么东西在拉扯这段关系（秘密/误解/利益冲突/身份差距）
3. **变化路径**：关系经历哪些转折，最终走向何方

**关系变化与大纲的对应**：

人物关系的转折点应该与大纲的幕间转折点呼应。这不是巧合，而是设计：

```
大纲turning_point: "主角发现搭档是卧底"
→ 关系变化: 从"并肩作战的伙伴" 转变为 "不知是敌是友的危险存在"
→ 这个关系转折直接驱动第二幕的剧情走向
```

### 5.3 多角色关系网

如果有多个重要角色，关系不是一对一的，而是一张网：

```
A-B: 表面合作，暗中互相利用
A-C: 师徒关系，A对C有愧疚
B-C: B嫉妒C得到A的偏爱

→ 三角关系中，每条关系的变化都会影响其他两条
→ B揭露A的秘密 → A-C信任崩塌 → C转向B → A被孤立
```

## 6. 常见问题自查

| 问题 | 症状 | 解决方法 |
|---|---|---|
| traits同质化 | 5个traits说的是一件事 | 表层+深层分组，确保有矛盾点 |
| 动机脱节 | 显性隐性动机没有张力 | 检查：达成显性目标后，隐性需求是否仍未满足？ |
| 恐惧太抽象 | "害怕失败"这种万能恐惧 | 具体到场景：在什么情况下、以什么方式失败 |
| 配角无名 | 用泛称如"系统管理员""副本BOSS" | 必须给具体名字或固定称谓，否则正文中无法识别 |
| 配角工具人 | 配角只在需要时出现 | 给配角自己的弱点和诉求 |
| 关系静态 | 从头到尾关系没变化 | 设计至少2个关系转折点 |
| 人物孤岛 | 角色之间没有交叉影响 | 检查关系网，确保变化会连锁反应 |
